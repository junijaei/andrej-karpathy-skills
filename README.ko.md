# Karpathy에서 영감받은 Claude Code 가이드라인

> 제 새 프로젝트 [Multica](https://github.com/multica-ai/multica)를 확인해보세요 — 재사용 가능한 스킬 기반으로 코딩 에이전트를 실행하고 관리하는 오픈소스 플랫폼입니다.
>
> X에서 팔로우: [https://x.com/jiayuan_jy](https://x.com/jiayuan_jy)

[Andrej Karpathy가 관찰한 LLM 코딩의 문제점](https://x.com/karpathy/status/2015883857489522876)을 바탕으로, Claude Code의 행동 방식을 개선하기 위해 작성된 단일 `CLAUDE.md` 파일입니다.

[English](./README.md) | [简体中文](./README.zh.md) | 한국어

## 문제점

Andrej의 글에서 발췌:

> "모델은 사용자를 대신해 잘못된 가정을 하고도 확인 없이 그냥 진행해 버립니다. 불확실한 부분을 방치하고, 명확한 설명을 요구하지도 않으며, 모순점을 드러내거나 트레이드오프를 제시하지도 않고, 반박해야 할 때에도 그냥 넘어갑니다."

> "모델은 코드와 API를 지나치게 복잡하게 만들고, 추상화를 불필요하게 부풀리며, 죽은 코드를 정리하지 않습니다... 100줄이면 충분할 코드를 1000줄이 넘는 방대한 구조로 구현하기도 하죠."

> "작업과 무관한 코드이더라도, 자신들이 제대로 이해하지 못한 주석이나 코드를 변경하거나 제거하는 일이 여전히 있습니다."

## 해결책

이러한 문제들을 즉시 해결하기 위한 한 파일 안의 4가지 원칙입니다:

| 원칙 | 해결하는 문제 |
|-----------|-----------|
| **코딩 전에 생각하기** | 잘못된 가정, 숨겨진 혼란, 누락된 트레이드오프 |
| **단순함 우선** | 과도한 복잡성, 부풀려진 추상화 |
| **최소 침습적 변경** | 불필요한 수정, 건드려선 안될 코드 수정 |
| **목표 주도 실행** | 테스트 우선 방식, 검증 가능한 성공 기준의 활용 |

## 네 가지 원칙 상세 설명

### 1. 코딩 전에 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 표면화할 것.**

LLM은 종종 조용히 하나의 해석을 선택하고 그대로 진행합니다. 이 원칙은 명시적인 추론을 강제합니다:

- **가정을 명확히 밝히기** — 불확실하면 추측하지 말고 질문한다
- **여러 해석을 제시하기** — 모호하다면 임의로 하나를 선택하지 않는다
- **필요할 때 반박하기** — 더 간단한 방법이 있다면 말한다
- **혼란스러울 땐 멈추기** — 무엇이 불명확한지 명시하고 설명을 요구한다

### 2. 단순함 우선

**문제를 해결하는 최소한의 코드만. 추측성 코드 금지.**

과도한 엔지니어링 경향에 맞섭니다:

- 요청받지 않은 기능은 추가하지 않는다
- 일회용 코드를 위한 추상화는 하지 않는다
- 요청받지 않은 "유연성"이나 "설정 가능성"을 추가하지 않는다
- 발생 불가능한 시나리오에 대한 에러 처리는 하지 않는다
- 200줄의 코드를 50줄로 줄일 수 있다면 다시 작성한다

**테스트:** 시니어 엔지니어가 이것을 보고 지나치게 복잡하다고 말할까요? 그렇다면 단순화하세요.

### 3. 최소 침습적 변경

**반드시 필요한 부분만 건드릴 것. 자신이 만든 것만 정리할 것.**

기존 코드를 수정할 때:

- 인접한 코드, 주석 또는 포맷팅을 "개선"하지 않는다
- 고장 나지 않은 것을 리팩토링하지 않는다
- 다른 방식을 선호하더라도 기존 스타일을 따른다
- 관련 없는 죽은 코드를 발견했다면, 언급은 하되 삭제하지 않는다

변경 사항으로 인해 고립된(orphan) 코드가 발생한 경우:

- 자신의 변경으로 인해 사용되지 않게 된 import/변수/함수만 제거한다
- 요청받지 않은 이상 기존의 죽은 코드는 건드리지 않는다

**테스트:** 변경된 모든 줄은 사용자의 요청과 직접적으로 연결되어야 합니다.

### 4. 목표 주도 실행

**성공 기준을 정의할 것. 검증될 때까지 반복할 것.**

명령형 작업을 검증 가능한 목표로 전환합니다:

| 기존 지시 | 변환된 지시 |
|--------------|-----------------|
| "유효성 검사 추가해 줘" | "잘못된 입력에 대한 테스트를 작성한 후 통과시키기" |
| "버그 고쳐 줘" | "버그를 재현하는 테스트를 작성한 후 통과시키기" |
| "X 리팩토링해 줘" | "리팩토링 전후로 테스트가 통과되는지 확인하기" |

여러 단계의 작업인 경우 간단한 계획을 명시합니다:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준은 LLM이 독립적으로 반복 작업을 수행할 수 있게 합니다. 모호한 기준("되게 만들어줘")은 지속적인 재확인을 필요로 합니다.

## 설치

**옵션 A: Claude Code 플러그인 (권장)**

Claude Code 내에서 먼저 마켓플레이스를 추가하세요:
```
/plugin marketplace add forrestchang/andrej-karpathy-skills
```

그 다음 플러그인을 설치하세요:
```
/plugin install andrej-karpathy-skills@karpathy-skills
```

이렇게 하면 가이드라인이 Claude Code 플러그인으로 설치되어 모든 프로젝트에서 해당 스킬을 사용할 수 있습니다.

**옵션 B: CLAUDE.md (프로젝트별)**

새 프로젝트:
```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

기존 프로젝트 (하단에 추가):
```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

## Cursor에서 사용하기

이 레포지토리에는 커밋된 Cursor 프로젝트 규칙([.cursor/rules/karpathy-guidelines.mdc](.cursor/rules/karpathy-guidelines.mdc))이 포함되어 있어, Cursor에서 프로젝트를 열 때도 동일한 가이드라인이 적용됩니다. 설정 방법, 다른 프로젝트에서의 규칙 사용법, 그리고 Claude Code와 어떻게 관련되는지에 대해서는 **[CURSOR.md](CURSOR.md)**를 참조하세요.

## 핵심 인사이트

Andrej의 글에서:

> "LLM은 특정 목표를 달성할 때까지 반복하는 데 탁월합니다... 무엇을 해야 할지 알려주지 말고, 성공 기준을 제시한 뒤 지켜보세요."

"목표 주도 실행" 원칙이 바로 이것을 담고 있습니다. 명령형 지시를 검증 루프가 있는 선언적 목표로 전환합니다.

## 잘 작동하는지 확인하는 방법

다음과 같은 현상이 보인다면 가이드라인이 잘 작동하고 있는 것입니다:

- **diff에서 불필요한 변경 감소** — 요청한 변경 사항만 나타납니다
- **과도한 복잡성으로 인한 재작성 감소** — 처음부터 코드가 단순합니다
- **구현 전 명확한 질문 발생** — 실수한 후가 아니라, 작업 전에 명확히 묻습니다
- **깔끔하고 최소화된 PR** — 불필요한 리팩토링이나 "개선 사항"이 없습니다

## 커스터마이징

이 가이드라인은 프로젝트별 지침과 병합되도록 설계되었습니다. 기존 `CLAUDE.md`에 추가하거나 새 파일을 만드세요.

프로젝트별 규칙의 경우, 다음과 같이 섹션을 추가하세요:

```markdown
## Project-Specific Guidelines

- Use TypeScript strict mode
- All API endpoints must have tests
- Follow the existing error handling patterns in `src/utils/errors.ts`
```

## 트레이드오프 참고 사항

이 가이드라인은 **속도보다 안정성**을 우선시합니다. 단순한 오타 수정이나 명백한 한 줄짜리 코드 같은 사소한 작업의 경우 재량껏 판단하세요. 모든 변경 사항에 엄격한 기준을 적용할 필요는 없습니다.

목표는 중요한 작업에서의 값비싼 실수를 줄이는 것이지, 단순한 작업을 지연시키는 것이 아닙니다.
